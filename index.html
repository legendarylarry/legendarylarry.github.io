Machine Learning Example

A simple pipeline which reads millions of records containing MS Windows system specifications and scores their likelihood of being infected with malware. 



windows_defender_ingest:
    extract:
      type: fs_csvs
      root: surge/ml/microsoft_vulnerability/data
      poll: 3
      batch_size: 14
    transformers:
      - type: none
    load:
      type: kafka
      serializer: json
      topic: microsoft_vulnerabilities_extract
      connection: *kafka_connection
             

defender_scoring:
    extract:
      type: kafka
      deserializer: json
      topic: microsoft_vulnerabilities_extract
      connection: *kafka_connection
      poll: 3

    transformers:
      - type: api_tag
        connection:
          url: http://localhost:9102/predict
        mappings:
          - source: 0
            destination: "HasDetections"

    load:
      type: kafka
      serializer: json
      topic: microsoft_vulnerabilities_transform
      connection: *kafka_connection
             

defender_load_warehouse:
    extract:
      type: kafka
      deserializer: json
      topic: microsoft_vulnerabilities_transform
      connection: *kafka_connection
      poll: 3

    transformers:
      - type: none
    load:
      type: elasticsearch
      index: microsoft_vulnerability
      connection: *elk_connection

The ETL subsystem itself can be secured in several ways.

TLS enabled on all transfers.

Encryption at rest (if warranted).

Authentication/Authorization gates on Message Busses and Datastores.

Enterprise Security

Data aggregation enhances security of the enterprise.

Collection of SIEM data from IT tools.

Event-Based automated reactions to detected anomalies.

Scale Horizontally

Identical ETL nodes enable seamless horizontal scaling in a cloud environment without additional engineering.

Scalable Ingest

Extractor nodes pull data from an upstream source.

The extractor nodes are clones, to add more ingest capacity, add more nodes or vertically scale individual node resources.

Cost Efficiency

Balancing node power and number is an exercise in maximizing cost efficiency. More powerful nodes are more cost effective for sustained throughput, while smaller nodes which grow/shrink in number are more cost effective for handling bursts and lulls.

We perform a quantitative analysis of data throughput to determine the most cost-effective implementation, but the flexibility of design means that pivoting to a new strategy is simple and quick.

Scalable Distribution

Distribution nodes can likewise be simply scaled to meet needs. We place them behind a load balancer to retain a single access point for consumers.

Data Streams

Leveraging the same data stream configuration and surge ETL library used for data ingest, streams can move data from Near-Real-Time access locations, to general data access storage, and finally to long-term archival (cold) storage, based on configurable timelines and data retention policies.

Cold Storage Stream Example


    to_cold_archive:
     extract:
       type: filesystem
       poll: 30min
       locations:
          modis:
            directory: /var/data/modis
            filter: *\.hdf5?
            ttl: 1year
            recursive: true
     transform:
       type: none
     load:
       type: aws_glacier
       vault: my-cold-storage
       connection: *aws_connection

Data Catalog

Collected and generated datasets are treated like software products with a development lifecycle, semantic versioning, release notes, planned deprecation schedules, and tracked consumption metrics.

Common Metadata Schema

Inspired by our work on the NASA EOSDIS Unified Metadata Model, we assess both short and long-term data ingest and data production interests, to develop a metadata schema which will contain metadata from disparate datasets in a common format. This common metadata schema enables the creation of a powerful data catalog which can then be levereged for many useful purposes.

Cross-Dataset Discovery and Aggregate Metrics

With a common metadata schema, search across datasets is possible since the datasets adhering to the schema have common indexable fields. Additionally, properties of the entire collection can be shown, e.g. temporal range, spatial extent, target interest/keywords, granule formats, further processing capabilities, documentation linkage, average granule size, resolution, etc.

Data I/O : Stream-Based Configuration

Streams are the atomic configuration unit in Innovisions. A pipeline can contain many data streams, and the process of moving data from one point to another may utilize multiple streams. Each stream contains an Extract, Transform, and Load definition corresponding to the input data source, operations on the data, and the output destination.


Extract

Extractors in the surge ETL library are python classes which implement both a fetch() method, and a means to track progress so that subsequent executions can begin from the current position. Consuming a new API or data source only requires creating a new Extractor class and implementing the fetch() method.

Transform

Transformers receive records from Extractors and use or modify the records before passing the results to a Loader for storage. Transformers can convert or restructure data formats, subset data for particular interesting fields, or combine multiple inputs to generate composite results.

Load

Loaders take the results of Transformers and store them. Loaders currently exist for multiple database types but also can outputt o filesystems, AWS S3, or other receiving APIs. Adding additional Loaders only requires creating a python class for the loader which implements the post(records: list) method.
